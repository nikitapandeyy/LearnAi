01 — Foundation
Pydantic Basics
The complete guide to Python's most powerful data validation library — what it is, why it matters, and how it works internally.
🧩
What is Pydantic?
foundation
▾

Pydantic is a Python library that uses type hints to validate, parse, and serialize data automatically. Think of it as a smart gatekeeper — before data enters your system, Pydantic checks it, converts it, and raises errors if anything is wrong.

Before Pydantic, developers wrote manual checks like if not isinstance(data['id'], int): raise ValueError(...). Pydantic eliminates all of that.

💡 Real-World Analogy
Pydantic is like a strict hotel receptionist. You must fill out the check-in form correctly — name (string), room number (int), check-in date (date). Wrong type? The receptionist converts it or rejects it. You can't sneak in bad data.
⚙️

Automatic Type Conversion (Coercion)
core concept
▾

Pydantic doesn't just check types — it converts them when possible. This is called coercion.

Python
# You pass a string, Pydantic gives you an int
user = User(id="10", name="nikita")
print(user.id)  # 10  ← not "10", it's an integer!
print(type(user.id))  # <class 'int'>
⚠️ Common Mistake
Coercion is not unlimited. User(id="hello") will FAIL — Pydantic cannot convert "hello" to an integer. Only logically convertible values work: "10" → 10, 1 → True, etc.

Complete Employee Model Example
real code
▾
Python
from pydantic import BaseModel, Field
from typing import Optional

class Employee(BaseModel):

    name: str = Field(
        ...,                    # required
        min_length=3,
        max_length=30,
        description="Employee Name",
        example="Nikita"
    )

    dept: Optional[str] = "General"  # optional with default

    salary: int = Field(
        ...,
        ge=10000,         # must be ≥ 10,000
        le=1000000        # must be ≤ 10,00,000
    )

    phone: str = Field(
        ...,
        pattern=r'^\d{10}$'  # exactly 10 digits
    )

# Valid:
e = Employee(name="Nikita", salary=50000, phone="9876543210")

# Invalid — raises ValidationError:
Employee(name="ab", salary=5000, phone="123")
💡 Why ... (Ellipsis)?
The three dots ... is Python's "Ellipsis" object. In Pydantic's Field(...), it signals "this field is required — no default value". It's Pydantic's convention, not standard Python.

field_validator — 3 Real Examples
code deep dive
▾

Example 1 — Name Capitalization

Python
from pydantic import BaseModel, field_validator

class Person(BaseModel):
    f_name: str
    l_name: str

    @field_validator('f_name', 'l_name')  # field names as STRINGS!
    def must_be_capitalized(cls, v):
        if not v.istitle():
            raise ValueError("name must start with capital")
        return v  # ALWAYS return v!

# Valid: Person(f_name="Nikita", l_name="Pandey")
# Invalid: Person(f_name="nikita", ...)

Example 2 — Email Normalization (Transform Data)

Python
class User(BaseModel):
    email: str

    @field_validator('email')
    def normalize_email(cls, v):
        return v.lower().strip()
        # "  Nikita@GMAIL.COM " → "nikita@gmail.com"

# Validators can TRANSFORM data, not just validate!

Example 3 — Price Parsing (Type Conversion)

Python
class Product(BaseModel):
    price: str

    @field_validator('price')
    def parse_price(cls, v):
        if isinstance(v, str):
            return float(v.replace('$', '').replace(',', ''))
        return v
        # "$10,000" → 10000.0
🚨 Critical Rule — ALWAYS return v
If you forget return v, Python returns None implicitly. Pydantic then stores None as the field value. This is a silent bug — no error, just wrong data!

model_validator — Cross-Field Logic
advanced
▾
Python
from pydantic import BaseModel, model_validator

class Passwd(BaseModel):
    curr_pswd: str
    knw_passwd: str

    @model_validator(mode='after')  # runs AFTER all fields validated
    def check_passwords_match(cls, values):
        if values.curr_pswd != values.knw_passwd:
            raise ValueError("passwords do not match")
        return values  # return the full model, not a single field!

# mode='before' → runs before field validation
# mode='after'  → runs after all fields validated (most common)
ParameterMeaning
clsThe class itself (like self for class methods)
values (in model_validator)The entire model object with all fields
v (in field_validator)The single field's value
mode='after'Run validator after type checking is done

05 — Dynamic Fields
Computed Fields
Generate field values dynamically from other fields — and include them in serialized output — without storing them manually.
🧮
Hotel Billing Example
full example
▾
Python
from pydantic import BaseModel, computed_field

class Hotel(BaseModel):
    name: str
    night: int
    price: int

    @computed_field         # include in model_dump()
    @property               # access like attribute, not method call
    def total_price(self) -> float:
        return self.night * self.price

user = Hotel(name="Nikita", night=3, price=10000)

print(user.total_price)    # 30000
print(user.model_dump())   # includes 'total_price': 30000

06 — Structure
Nested Models
Models inside models — perfect for complex real-world data like users with addresses, orders with items, and APIs with structured bodies.
🏠
User with Address Example
full example
▾
Python
from pydantic import BaseModel

class Address(BaseModel):
    street: str
    pst: str
    state: str

class User(BaseModel):
    name: str
    phone: int
    adr: Address   # nested model

# Method 1: Pass an Address object directly
adrs = Address(street="value1", pst="7887", state="up")
user1 = User(name="nikita", phone=788778, adr=adrs)

# Method 2: Pass a plain dict — Pydantic auto-converts it!
user_data = {
    'name': 'nikita', 'phone': '45343',  # phone as string → auto-converts
    'adr': {'street': 'effe', 'pst': 'frwf', 'state': 'hu'}
}
user2 = User(**user_data)  # dict auto-becomes Address object!
💡 Superpower — Automatic Nested Parsing
You don't need to manually create Address(...) for user2. Pydantic sees the dict, sees the field type is Address, and converts the dict automatically. This is extremely useful when parsing JSON API responses!

from typing import List, Optional

Why Quotes? Forward References Explained
deep explanation
▾
🧠 Why "Comment" in Quotes?
When Python is reading the class definition of Comment, it hasn't finished defining it yet. If you write List[Comment], Python tries to resolve Comment immediately — but it doesn't exist yet! Writing List["Comment"] as a string tells Python: "I'll resolve this later." This is called a Forward Reference.
🧠 Why model_rebuild()?
After the class is fully defined, model_rebuild() tells Pydantic: "Now go resolve those forward reference strings into actual types." Without it, Pydantic may not correctly understand the recursive structure.
ValueMeaning
reply = NoneNo replies field provided at all
reply = []Field exists but is an empty list
reply = [Comment(...)]Has actual replies

Custom Datetime Formatting with ConfigDict
advanced
▾
Python
from pydantic import BaseModel, ConfigDict
from datetime import datetime

class User(BaseModel):
    name: str
    createdAt: datetime

    model_config = ConfigDict(
        json_encoders={
            datetime: lambda v: v.strftime('%d-%m-%Y %H:%M:%S')
            # datetime object → "03-04-2024 02:13:02"
        }
    )

user = User(name="nikita", createdAt=datetime(2024,4,3,2,13,2))
print(user.model_dump_json())
# {"name":"nikita","createdAt":"03-04-2024 02:13:02"}
